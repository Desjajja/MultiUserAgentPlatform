用途：为 Arize Phoenix 提供可按业务维度过滤的 trace/span 元数据

适用范围：所有 host-side 和 container-side observability spans

2. 业务标签注册表（Business Tag Registry）

所有业务维度作为 key 写入单个 metadata JSON 对象，再编码为 span 的 metadata attribute。
内部 key 不带 muap. 前缀（前缀语义由 metadata 命名空间本身承担）。

2.1 核心标签（所有 span 必须携带）

metadata key类型必填合法值说明
layerstring✅ai / platform区分 AI 语义与平台遥测

2.2 路由标签（Routing Spans）

metadata key类型必填合法值说明
route_typestring✅frontdesk / worker / erp / system路由类型（erp 当前 reserved，无运行时发射点）
lanestring✅frontdesk / worker业务车道
channelstring✅cli / feishu / webhook / unknown入口渠道
intentstring✅chat / approval / execute / system用户意图
route_type 内容绑定语义：route_type 绑定到内容相关 span（router.deliver_to_agent / agent.turn），代表"该路由下的实际回复内容"，而非单纯的路由动作。用户按 route_type 过滤时，期望看到的是该 lane 的真实对话内容。

2.3 会话标签（Session Spans）

metadata key类型必填合法值说明

3. 标签设置位置（Setting Locations）

下列各 span 的 metadata key 通过 applyBusinessTags(span, {...})（host）或 container 本地 helper 写入，最终编码为单个 metadata JSON attribute。 
router.deliver_to_agent (AGENT)
  metadata.layer = 'ai'
  metadata.route_type = 'frontdesk'   # 派活到 worker 时为 'worker'（按目标 lane 派生）
  metadata.lane = 'frontdesk'
  metadata.channel = 'cli' | 'feishu'
  metadata.intent = 'chat'
  metadata.agent_group = 'Demo'
  metadata.session_mode = 'shared'
  metadata.engage_mode = 'direct' | 'a2a'

agent.turn (CHAIN, container-side)
  metadata.layer = 'ai'
  metadata.route_type = <FRONTLANE_ROUTE_TYPE 派生>   # 不再 hardcode 'worker'
  metadata.lane = <派生>
  metadata.provider = 'sdk-openai' | 'claude'
  # session.id 为 OpenInference 语义字段（真实 attribute），由 FRONTLANE_SESSION_ID 注入，不再为 'unknown'

container.wake | container.spawn | container.kill
  metadata.layer = 'platform'
  metadata.route_type = <按 session lane 派生>
  metadata.lane = <派生>
  metadata.provider = 'sdk-openai'（仅 spawn）

delivery.session.drain | delivery.message.deliver | delivery.channel.send
  metadata.layer = 'platform'
  metadata.route_type = 'worker'
  metadata.lane = 'worker'

channel.cli.receive | channel.feishu.receive
  metadata.layer = 'platform'
  metadata.route_type = 'frontdesk'
  metadata.lane = 'frontdesk'
  metadata.channel = 'cli' | 'feishu'

router.route
  metadata.layer = 'platform'
  metadata.route_type = 'frontdesk'
  metadata.lane = 'frontdesk'
  metadata.channel = event.channelType
OpenInference 语义字段保留为真实 attribute（不进 metadata）：session.id、agent.group.id、output.value、output.mime_type、openinference.span.kind 等是 OpenInference 标准语义字段，继续作为普通 OTel attribute 写入，不放入 metadata。

4. Phoenix 过滤语法参考（Phoenix 17.2.0）

关键点：业务维度存储在 metadata 中（Phoenix 一等公民过滤字段）。三路过滤（UI / REST / SQL）均可用。

4.1 UI Filter Bar（Spans 标签页）

使用 Python 布尔表达式，直接用 metadata["key"] 访问：

按 layer 过滤
metadata["layer"] == "ai"
按路由类型过滤（看 frontdesk 实际回复内容）
metadata["route_type"] == "frontdesk"
按业务车道过滤
metadata["lane"] == "worker"
按渠道过滤
metadata["channel"] == "cli"
按 provider 过滤
metadata["provider"] == "sdk-openai"
组合过滤（AND 关系）
metadata["layer"] == "ai" and metadata["route_type"] == "frontdesk"
按 a2a 派活过滤（engage_mode 是 a2a 的唯一承载）
metadata["engage_mode"] == "a2a"
按 OpenInference kind 过滤（必须大写）
span_kind == "AGENT"
按 span 名称过滤
name == "agent.turn"
Phoenix 官方：metadata["topic"] == 'programming' 是文档示例；"Any expressions that work with the .where() method can also be used in the UI"。

4.3 PostgreSQL 直接查询

-- 查 AI 语义层 spans
SELECT name, span_kind, start_time
FROM spans
WHERE attributes->'metadata'->>'layer' = 'ai'
ORDER BY start_time DESC
LIMIT 20;

-- 按路由类型 + 渠道组合
SELECT name, span_kind, attributes->'metadata'->>'channel' AS channel
FROM spans
WHERE attributes->'metadata'->>'route_type' = 'frontdesk'
  AND attributes->'metadata'->>'channel' = 'cli'
ORDER BY start_time DESC;

-- 查所有已设置 metadata 业务维度的 spans
SELECT
  name,
  span_kind,
  attributes->'metadata'->>'layer' AS layer,
  attributes->'metadata'->>'route_type' AS route_type,
  attributes->'metadata'->>'provider' AS provider
FROM spans
WHERE attributes->'metadata' IS NOT NULL
ORDER BY start_time DESC
LIMIT 50;
# 一行命令查询
docker exec muap-observability-prod-postgres-1 psql -U phoenix -d phoenix -c \
  "SELECT name, span_kind, attributes->'metadata'->>'layer' AS layer FROM spans \
   WHERE attributes->'metadata'->>'layer' = 'ai' ORDER BY start_time DESC LIMIT 10;"
注意：metadata 在 OpenInference 中被编码为 JSON 字符串 attribute（getMetadataAttributes 产出 { metadata: JSON.stringify(...) }）。Phoenix 17.x 摄取后在 Postgres 中以 JSONB 形式可用 attributes->'metadata'->>'key' 访问；若实际存储为字符串，需先 (attributes->>'metadata')::jsonb->>'key'。以 staging 实测为准。

5. 实现机制（为什么这样编码）

5.1 host 端：getMetadataAttributes

import { getMetadataAttributes } from '@arizeai/openinference-core';
// applyBusinessTags 内部：
span.setAttributes(getMetadataAttributes({ layer: 'ai', route_type: 'frontdesk', ... }));
// 等价于 span.setAttribute('metadata', JSON.stringify({ layer:'ai', route_type:'frontdesk', ... }))
  • host 已装 @arizeai/openinference-core@2.2.0，无需新依赖。
  • applyBusinessTags(span, {...}) 调用面不变，15 个 host emitter 零改动。

5.2 container 端：本地 helper

  • container（container/agent-runner/）无 @arizeai 依赖，刻意不引入以避免镜像膨胀。
  • 使用本地 helper（container/agent-runner/src/observability/metadata.ts）输出等价 { metadata: JSON.stringify(record) }。

5.3 为什么不用 setMetadata / using_metadata

  • setMetadata 只把数据放进 OTel context，需 OITracer / auto-instrumentation 在 startSpan 时读取并落到 span。
  • 本项目业务 span 全部用裸 @opentelemetry/api 的 tracer.startSpan() / startActiveSpan()，context metadata 永远不会落到 span。
  • 因此必须用 getMetadataAttributes（host）/ 本地 helper（container）显式写入 metadata attribute。
  • 迁移到 OITracer 是 future work（见 ADR-0017）。

5.4 metadata 合并注意

  • metadata 是单个 JSON attribute。同一 span 多次写入业务维度时必须 union，不能后写覆盖先写。
  • 实现上应累积所有维度后一次性编码，或 read-merge-rewrite。staging 需实测 span 最终携带维度并集。

6. 已知限制（Phoenix 17.2.0）

限制状态说明
attributes 作为 UI 过滤字段❌ 不支持UI filter bar 不接受 attributes 作为一等过滤字段；这正是迁移到 metadata 的根因
Traces 标签页过滤❌ 仅根 spanTraces 视图只能按根 span 过滤；查子 span 用 Spans 视图
列表类型 metadata 值❌ 不支持简单 filtertags: ["a","b"] 无法用简单 filter 匹配
正则匹配❌ 不支持所有 filter 为精确匹配
