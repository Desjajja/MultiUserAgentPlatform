MUAP LiteLLM Proxy 迁移方案

可行性分析与实施计划 — 2026-06-01

1. 执行摘要

将 LiteLLM Proxy 作为 LLM 网关接入 MUAP 平台，实现：

  • 统一 LLM 可观测性 — token 用量、成本、延迟自动出现在 Phoenix trace 中
  • 补齐 PR-O3 容器侧 OTel 缺口 — ADR-0011 已记录的待办项
  • 为未来多模型路由/fallback/限流打基础

2.3 baseUrl 配置流

Host process.env.OPENAI_BASE_URL
  → src/providers/openai.ts (buildOpenAIContribution)
  → docker run -e OPENAI_BASE_URL=...
  → Container process.env
  → openai.ts: normalizeBaseUrl() → this.baseUrl
  → fetch(`${this.baseUrl}/responses`)
结论：零代码改动即可重定向 LLM 流量，只需设置 OPENAI_BASE_URL=http://host.docker.internal:4000

6. 风险评估

风险概率影响缓解
LiteLLM 不兼容某些 Responses API 参数低中我们用 stream:false，已验证兼容
容器 OTel 初始化增加启动延迟低低OTel SDK init < 50ms
LiteLLM Proxy 宕机导致 LLM 不可用中高健康检查 + fallback 到直连
trace_id 不匹配导致 Phoenix 显示断裂低中端到端验证
Docker 网络问题（host.docker.internal 不通）低高已有 hostGatewayArgs() 处理 Linux

7. 回退策略

  1. 移除 OPENAI_BASE_URL 覆盖（或设回 https://api.openai.com/v1）
  2. 容器 OTel 代码保留（不影响功能，只是 trace 不经过 proxy）
  3. LiteLLM compose stack 可独立停止：pnpm litellm:down

8. 收益总结

收益说明
统一 LLM 可观测性每次 LLM 调用自动出现在 Phoenix trace，含 token/cost/latency
补齐 PR-O3容器侧 OTel 初始化 + traceparent 传播一并完成
完整 trace treeHost → Container → LiteLLM → 全部在一个 Phoenix trace
模型路由/fallbackLiteLLM 支持 model alias、fallback chain、load balancing
成本追踪gen_ai.cost.* 自动计算
Rate limit 管理LiteLLM 内置 RPM/TPM 限流
多模型统一接口未来加新 provider 只需改 LiteLLM config

9. 审批签章

请确认以下决策项

1. LiteLLM Proxy 作为独立 compose stack（muap-litellm），不合并到 observability stack
2. 仅 openai / sdk-openai / codex provider 经过 LiteLLM，claude provider 保持原样
3. 容器 OTel 采用最小化方案（仅 trace propagation，不做 full auto-instrumentation）
4. 使用 host.docker.internal:4000 作为容器到 proxy 的通信路径
5. 向后兼容：不设 LITELLM_PROXY_URL 时系统照常直连 OpenAI
